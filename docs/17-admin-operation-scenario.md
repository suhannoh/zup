# 17. 관리자 운영 시나리오

## 1. 목적

이 문서는 Zup 관리자 화면에서 실제 운영 데이터를 입력하고 검수 흐름을 확인하는 시나리오를 정리한다.

운영자는 이 문서를 따라 브랜드, 혜택, 공식 출처, 태그, 검수 이력, 사용자 제보 흐름을 점검할 수 있다.

## 2. 시나리오 1: 새 브랜드 등록

경로:

```text
/admin/brands
```

절차:

1. 브랜드 관리 화면으로 이동한다.
2. 브랜드 등록 폼을 연다.
3. 카테고리를 선택한다.
4. 브랜드명을 입력한다.
5. 브랜드 slug를 입력한다.
6. 공식 홈페이지 URL을 입력한다.
7. 멤버십 URL 또는 앱 URL이 있으면 입력한다.
8. 활성 상태를 확인한다.
9. 저장한다.
10. 브랜드 목록에 새 브랜드가 표시되는지 확인한다.

확인 기준:

- 브랜드명과 slug가 중복되지 않는다.
- 카테고리가 올바르다.
- 공식 URL이 실제 공식 채널이다.
- 활성 상태가 의도한 값이다.

## 3. 시나리오 2: 새 혜택 등록

경로:

```text
/admin/benefits
```

절차:

1. 혜택 관리 화면으로 이동한다.
2. 혜택 등록 폼을 연다.
3. 브랜드를 선택한다.
4. 혜택명과 요약을 입력한다.
5. 혜택 유형을 선택한다.
6. `occasionType`은 MVP 기준 `BIRTHDAY`로 둔다.
7. 생일 사용 기간 유형을 선택한다.
8. 앱 필요 여부, 멤버십 필요 여부, 구매 조건 여부를 입력한다.
9. 공식 출처 확인 전이면 `DRAFT` 상태로 저장한다.
10. 저장 후 혜택 목록에 표시되는지 확인한다.

확인 기준:

- 공식 출처가 아직 없으면 `PUBLISHED`로 저장하지 않는다.
- 조건이 모호하면 `NEEDS_CHECK` 또는 `DRAFT`로 둔다.

## 4. 시나리오 3: 공식 출처 등록

경로:

```text
/admin/benefits/{id}
```

절차:

1. 혜택 카드에서 `출처/태그/이력 관리`로 이동한다.
2. 공식 출처 등록 영역을 확인한다.
3. 출처 유형을 선택한다.
4. `sourceUrl`을 입력한다.
5. 출처 제목을 입력한다.
6. `sourceCheckedAt`을 입력한다.
7. 필요하면 메모를 남긴다.
8. 저장한다.
9. 출처 목록에 등록된 출처가 표시되는지 확인한다.

확인 기준:

- URL이 공식 채널이다.
- 확인일이 실제 검수일이다.
- 블로그/커뮤니티 출처는 힌트로만 사용한다.

## 5. 시나리오 4: 태그 연결

경로:

```text
/admin/benefits/{id}
```

절차:

1. 태그 관리 영역을 확인한다.
2. 전체 태그 select에서 연결할 태그를 선택한다.
3. `무료`, `앱 필요`, `생일월` 등 실제 조건과 일치하는 태그를 연결한다.
4. 중복 태그가 연결되지 않는지 확인한다.
5. 필요 없는 태그는 삭제한다.

확인 기준:

- 혜택 조건과 태그가 일치한다.
- 확인되지 않은 조건은 태그로 단정하지 않는다.
- 모호한 경우 `확인 필요` 태그를 사용한다.

## 6. 시나리오 5: 게시 처리

경로:

```text
/admin/benefits
```

절차:

1. 혜택 관리 화면으로 이동한다.
2. 대상 혜택의 검수 상태 변경 폼을 확인한다.
3. 검수 상태를 `PUBLISHED`로 변경한다.
4. `lastVerifiedAt`을 입력한다.
5. 검수 메모를 입력한다.
6. 저장한다.
7. 사용자 화면 `/brands/{slug}`로 이동한다.
8. 해당 혜택이 노출되는지 확인한다.

확인 기준:

- 공식 출처가 등록되어 있다.
- 혜택이 `PUBLISHED` 상태다.
- 혜택 `isActive=true`다.
- 브랜드가 활성 상태다.
- 사용자 화면에 혜택이 표시된다.

## 7. 시나리오 6: 사용자 제보 처리

제보 경로:

```text
/reports/new
```

관리 경로:

```text
/admin/reports
/admin/benefits/{id}
```

절차:

1. `/reports/new`에서 기존 혜택과 연결된 제보를 제출한다.
2. `WRONG_INFO`, `BENEFIT_ENDED`, `CONDITION_CHANGED`, `OFFICIAL_LINK_FOUND` 유형이면 관련 혜택이 `NEEDS_CHECK`로 전환되는지 확인한다.
3. `/admin/reports`에서 제보 목록을 확인한다.
4. 제보 상태를 `REVIEWING`, `RESOLVED`, `REJECTED` 중 운영 상황에 맞게 변경한다.
5. `adminMemo`에 처리 내용을 입력한다.
6. `/admin/benefits/{id}`에서 VerificationLog가 생성되었는지 확인한다.
7. 공식 출처를 재확인하고 혜택 정보를 수정한다.
8. 필요한 경우 다시 `PUBLISHED`로 변경한다.

확인 기준:

- 제보가 정상 생성된다.
- 관련 혜택이 `NEEDS_CHECK`로 전환된다.
- 중복 `NEEDS_CHECK` 상태에서는 VerificationLog가 중복 생성되지 않는다.
- 제보 처리 메모가 저장된다.
- 검수 이력이 혜택 운영 상세 화면에 표시된다.
