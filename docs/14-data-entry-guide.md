# 14. 운영 데이터 입력 가이드

## 1. 목적

이 문서는 Zup 관리자가 브랜드와 생일 혜택 데이터를 입력할 때 따라야 할 운영 기준을 정리한다.

Zup은 블로그식 혜택 모음이 아니라 공식 출처 기준으로 검수 가능한 생일 혜택 정보를 제공하는 서비스다. 따라서 실제 혜택을 입력할 때는 브랜드, 혜택, 출처, 태그, 검수 상태를 한 흐름으로 관리해야 한다.

## 2. 브랜드 등록 기준

브랜드는 다음 조건을 만족할 때 등록한다.

- 사용자가 브랜드명을 직접 검색할 가능성이 있다.
- 생일 혜택 존재 여부를 공식 출처로 확인할 수 있거나 확인 후보로 관리할 가치가 있다.
- 카테고리를 명확히 지정할 수 있다.
- 공식 홈페이지, 멤버십 안내, 앱 안내 중 하나 이상의 공식 URL을 확인할 수 있다.

브랜드 등록 시 최소 입력값:

- 카테고리
- 브랜드명
- 브랜드 slug
- 공식 홈페이지 URL 또는 멤버십 URL
- 활성 상태

MVP에서는 브랜드 로고 무단 사용을 피한다. `logoUrl`은 선택값이며, 공식 사용 권한이 확인되지 않은 로고 이미지는 입력하지 않는다.

## 3. 혜택 등록 기준

혜택은 브랜드에 연결된 단위 정보로 등록한다.

혜택 등록 시 최소 입력값:

- 브랜드
- 혜택명
- 요약
- 혜택 유형
- 대상 유형: MVP에서는 `BIRTHDAY`
- 생일 사용 기간 유형
- 앱 필요 여부
- 멤버십 필요 여부
- 구매 조건 여부
- 검수 상태
- 활성 상태

입력 중이거나 공식 출처가 아직 확인되지 않은 혜택은 `DRAFT` 또는 `NEEDS_CHECK` 상태로 둔다.

공식 출처가 없는 혜택은 `PUBLISHED`로 변경하지 않는다.

## 4. 공식 출처 등록 기준

혜택을 게시하려면 공식 출처를 최소 1개 이상 연결한다.

우선 등록할 출처:

- 공식 홈페이지
- 공식 멤버십 안내
- 공식 FAQ
- 공식 공지사항
- 공식 앱 내 쿠폰 또는 멤버십 화면
- 공식 SNS
- 고객센터 답변

블로그/커뮤니티 정보는 수집 힌트로만 사용한다. 게시 근거로 사용하지 않는다.

출처 등록 시 확인할 값:

- 출처 유형
- 출처 URL
- 출처 제목
- 확인일
- 메모

확인일은 운영자가 해당 출처를 실제로 확인한 날짜를 입력한다.

## 5. 태그 연결 기준

태그는 사용자가 조건별로 혜택을 찾을 수 있게 돕는 필터다.

태그 연결 예시:

- 무료
- 할인
- 쿠폰
- 앱 필요
- 앱 불필요
- 회원가입 필요
- 멤버십 필요
- 조건 없음
- 최소 구매 필요
- 생일 당일
- 생일월
- 생일 전후 7일
- 공식 확인
- 확인 필요

태그는 혜택의 실제 조건과 일치할 때만 연결한다. 조건이 모호하면 `확인 필요` 태그를 사용하고 혜택 상태를 `NEEDS_CHECK`로 둔다.

## 6. 검수 상태 변경 기준

검수 상태는 혜택의 운영 상태를 나타낸다.

| 상태 | 의미 | 사용 기준 |
| --- | --- | --- |
| `DRAFT` | 입력 중 | 기본 정보만 입력했거나 공식 출처 확인 전 |
| `NEEDS_CHECK` | 공식 확인 필요 | 사용자 제보가 있거나 조건이 모호한 상태 |
| `VERIFIED` | 공식 확인 완료 | 공식 출처로 확인했으나 아직 공개 전 |
| `PUBLISHED` | 사용자 공개 | 공식 출처 확인이 끝났고 사용자 화면에 공개할 상태 |
| `EXPIRED` | 종료 확인 | 혜택 종료가 공식적으로 확인된 상태 |
| `HIDDEN` | 숨김 | 운영상 노출하지 않을 상태 |

`PUBLISHED`로 변경할 때는 최근 확인일과 검수 메모를 남긴다.

## 7. PUBLISHED 처리 전 체크리스트

`PUBLISHED`로 변경하기 전에 아래 항목을 모두 확인한다.

- 브랜드가 활성 상태다.
- 혜택이 활성 상태다.
- 공식 출처가 1개 이상 등록되어 있다.
- 공식 출처 URL이 정상 접속된다.
- 혜택명과 요약이 사용자에게 오해 없이 읽힌다.
- 앱 필요 여부가 정확하다.
- 멤버십 필요 여부가 정확하다.
- 구매 조건 여부가 정확하다.
- 사용 가능 기간 설명이 입력되어 있다.
- 최근 확인일이 입력되어 있다.
- 태그가 실제 조건과 일치한다.
- 블로그/커뮤니티 정보만으로 작성된 내용이 아니다.

## 8. 사용자 화면 노출 조건

사용자 화면에는 `PUBLISHED` 상태이면서 `isActive=true`인 혜택만 노출된다.

따라서 운영자는 다음 두 조건을 모두 확인해야 한다.

- `verificationStatus = PUBLISHED`
- `isActive = true`

혜택이 사용자 화면에 보이지 않는다면 먼저 위 두 조건과 브랜드 활성 상태를 확인한다.

## 9. 기본 입력 순서

1. 브랜드를 등록한다.
2. 혜택을 `DRAFT` 상태로 등록한다.
3. 공식 출처를 등록한다.
4. 혜택 조건에 맞는 태그를 연결한다.
5. 공식 출처 기준으로 내용을 검수한다.
6. 검수 상태를 `VERIFIED` 또는 `PUBLISHED`로 변경한다.
7. `/brands/{slug}` 사용자 화면에서 노출을 확인한다.
8. 이후 사용자 제보가 들어오면 `NEEDS_CHECK` 전환 여부와 검수 이력을 확인한다.
