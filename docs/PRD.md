# PRD — Zup

## 1. 제품명

Zup

## 2. 제품 한 줄 정의

몰라서 못 받던 혜택, 오늘 줍자.

Zup은 생일에 받을 수 있는 브랜드별 무료 쿠폰, 할인, 멤버십 혜택을 공식 출처 기준으로 정리하고 조건별로 빠르게 확인할 수 있는 검색 유입형 웹서비스다.

브랜드명은 장기적으로 신규가입 혜택, 시즌 혜택, 멤버십 혜택까지 확장 가능하게 설계하지만, 1차 MVP 범위는 브랜드별 생일 혜택 큐레이션으로 제한한다.

## 3. 문제 정의

사용자는 생일 전후로 브랜드 혜택을 검색한다.

하지만 정보는 흩어져 있다.

- 브랜드 앱
- 공식 홈페이지
- 멤버십 FAQ
- 이벤트 공지
- 블로그
- 커뮤니티
- 인스타 저장형 콘텐츠

기존 검색 결과의 문제:

- 최신성 불확실
- 조건 비교 불가
- 공식 출처 없음
- 앱 필요 여부 확인 어려움
- 생일 당일/생일월/전후 기간 구분 어려움
- 블로그 글을 여러 개 열어야 함

## 4. 사용자

### 1차 사용자

생일 전후로 혜택을 챙기고 싶은 20~30대 사용자.

### 검색 상황

- “생일 혜택 뭐 있지?”
- “스타벅스 생일 쿠폰 어떻게 받아?”
- “올리브영 생일 혜택 있어?”
- “영화관 생일 혜택 모음 없나?”
- “앱 설치 없이 받을 수 있는 생일 혜택 없나?”

## 5. 핵심 가치

- 생일 혜택을 한 곳에서 확인
- 조건별로 필터링
- 공식 출처 기준 신뢰도 제공
- 최근 확인일 표시
- 블로그보다 빠르게 비교
- 로그인 없이 바로 사용

## 6. 제품 원칙

1. 생일 혜택에 집중한다.
2. 사용자는 로그인하지 않는다.
3. 개인 생일 입력은 MVP에서 받지 않는다.
4. 공식 출처 없는 정보는 게시하지 않는다.
5. 블로그/커뮤니티는 힌트로만 사용한다.
6. 혜택마다 최근 확인일을 표시한다.
7. 불확실한 정보는 NEEDS_CHECK 상태로 관리한다.
8. 자동 크롤링은 MVP에서 하지 않는다.
9. 브랜드 로고 사용을 최소화한다.
10. SEO 페이지를 핵심 자산으로 본다.

## 7. 핵심 기능

### 공개 기능

- 브랜드 목록
- 브랜드 상세
- 카테고리별 혜택
- 태그별 혜택
- 조건 필터
- 브랜드 검색
- 정보 수정 제보
- 공식 출처 링크
- 최근 확인일

### 관리자 기능

- 브랜드 등록/수정/비활성화
- 혜택 등록/수정/비활성화
- 출처 등록/수정
- 태그 연결
- 검수 상태 변경
- 제보 관리
- 오래된 데이터 확인
- 조회수 확인

## 8. MVP 범위

### 포함

- 브랜드 30개
- 카테고리 6개
- 태그 10개 이상
- 관리자 CRUD
- 제보 기능
- SEO 기본 설정
- 사이트맵
- robots.txt

### 제외

- 사용자 회원가입
- 사용자 생일 입력
- 개인 추천
- AI 추천
- WebSocket
- 알림톡
- 대량 자동 크롤링
- 브랜드 로고 대량 사용
- 결제
- 제휴 관리

## 9. 데이터 수집 정책

출처 우선순위:

1. 공식 홈페이지
2. 공식 멤버십 안내
3. 공식 FAQ
4. 공식 공지사항
5. 공식 앱 내 쿠폰/멤버십 화면
6. 공식 SNS
7. 고객센터 답변
8. 블로그/커뮤니티 힌트

블로그/커뮤니티 정보는 공식 출처로 확인하기 전까지 게시하지 않는다.

## 10. 성공 지표

### 개발 지표

- 브랜드 30개 등록
- 브랜드 상세 페이지 30개 생성
- 공식 출처 링크 30개 이상 등록
- 최근 확인일 표시
- 제보 기능 동작
- 관리자 CRUD 동작
- 사이트맵 생성

### 운영 지표

- Google Search Console 등록
- Naver Search Advisor 등록
- 검색 노출 키워드 확인
- 실제 유입 여부 확인
- 제보 발생 여부 확인
- 오래된 데이터 재검수 루틴 수행

## 11. 확장 방향

생일 혜택 이후:

1. 신규 가입 혜택
2. 앱 설치 혜택
3. 첫 구매 혜택
4. 멤버십 등급 혜택
5. 시즌 혜택
6. 기념일 혜택

단, 1차 MVP에서는 확장하지 않는다.

## 12. 리스크

### 정보 정확성

혜택 조건이 바뀔 수 있다.

대응:

- 최근 확인일
- 공식 출처 링크
- 제보 버튼
- 검수 상태
- 면책 문구

### 로고/상표

로고 사용은 리스크가 있다.

대응:

- 텍스트/이니셜 뱃지 사용
- 공식 서비스가 아님을 명시

### 운영 피로도

혜택 최신성 유지가 필요하다.

대응:

- 브랜드 30개로 시작
- 조회수 높은 브랜드 우선 검수
- 오래된 데이터 자동 표시
- 사용자 제보 활용

## 13. 포트폴리오 포인트

- 정보 큐레이션 서비스의 데이터 구조화
- 공식 출처 기반 신뢰도 관리
- 검수 상태와 VerificationLog
- 사용자 제보 기반 운영 플로우
- SEO-friendly URL
- 조회수 기반 인기 데이터
- 추후 변화 감지 엔진 재사용 가능성
