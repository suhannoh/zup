# 00. 프로젝트 개요 — Zup

## 1. 프로젝트 한 줄 정의

**Zup**은 생일에 받을 수 있는 브랜드별 무료 쿠폰, 할인, 멤버십 혜택을 공식 출처 기준으로 정리하고, 사용자가 조건별로 빠르게 필터링해서 볼 수 있는 검색 유입형 정보 큐레이션 웹서비스다.

서비스 카피:

```text
몰라서 못 받던 혜택, 오늘 줍자
```

Zup은 장기적으로 몰라서 못 받던 브랜드 혜택, 신규가입 혜택, 시즌 혜택, 멤버십 혜택까지 확장 가능한 이름이다. 다만 1차 MVP는 브랜드별 생일 혜택 큐레이션에 집중한다.

## 2. 왜 만드는가

생일 혜택 정보는 브랜드 앱, 멤버십 페이지, FAQ, 이벤트 공지, 블로그, 커뮤니티에 흩어져 있다.

사용자는 보통 아래처럼 검색한다.

- 생일 혜택
- 생일 쿠폰
- 스타벅스 생일 쿠폰
- 올리브영 생일 혜택
- 영화관 생일 혜택
- 카페 생일 혜택
- 생일 무료 혜택

하지만 기존 검색 결과는 대부분 블로그 글이나 커뮤니티 글이다.

문제는 다음과 같다.

- 최신성이 불확실하다.
- 브랜드별 조건을 비교하기 어렵다.
- 앱 필요 여부, 멤버십 필요 여부, 사용 가능 기간이 한눈에 보이지 않는다.
- 생일 당일만 가능한 혜택과 생일월 내내 가능한 혜택이 섞여 있다.
- 공식 출처가 없는 정보가 많다.

이 프로젝트는 블로그처럼 읽는 정보가 아니라, **조건별로 걸러보는 혜택 DB**를 목표로 한다.

## 3. 핵심 가치

### 사용자 가치

- 생일에 받을 수 있는 혜택을 빠르게 찾는다.
- 앱 설치 여부, 멤버십 조건, 사용 가능 기간을 한눈에 확인한다.
- 공식 출처와 최근 확인일을 보고 신뢰도를 판단한다.
- 친구에게 공유하기 쉽다.

### 운영 가치

- 브랜드별 혜택 정보를 공식 출처 기준으로 구조화한다.
- 최근 확인일과 검수 상태를 관리한다.
- 사용자 제보를 통해 오래된 정보를 갱신한다.
- SEO 기반 롱테일 페이지를 만든다.

### 포트폴리오 가치

- 단순 CRUD가 아니라, 정보 신뢰도 관리가 포함된 운영형 서비스다.
- 관리자 검수 흐름, 최근 확인일, 공식 출처, 제보, 조회수 집계, SEO 페이지 구조를 설명할 수 있다.
- 추후 기존 변화 감지 엔진을 공식 출처 변경 감지 모듈로 재사용할 수 있다.

## 4. 핵심 차별점

이 프로젝트의 차별점은 “많이 모았다”가 아니다.

핵심은 다음이다.

> 블로그처럼 읽는 게 아니라, 내가 받을 수 있는 혜택만 조건별로 빠르게 걸러본다.

예시 필터:

- 무료 혜택
- 할인 혜택
- 쿠폰 혜택
- 앱 필요
- 앱 불필요
- 멤버십 필요
- 조건 없음
- 생일 당일
- 생일월
- 생일 전후 7일
- 공식 확인 완료

## 5. 서비스 원칙

- 사용자는 로그인 없이 정보를 볼 수 있어야 한다.
- 개인 생일 입력 없이도 가치가 있어야 한다.
- 브랜드 혜택 정보는 공식 출처 기준으로 검수한다.
- 블로그/커뮤니티는 수집 힌트로만 사용한다.
- 자동 크롤링으로 바로 게시하지 않는다.
- 변경 감지는 자동화할 수 있지만, 반영은 관리자가 검수한다.
- 브랜드 로고는 MVP에서 사용하지 않거나 최소화한다.
- MVP는 생일 혜택에만 집중한다.

## 6. 1차 MVP 목표

1차 MVP는 아래 상태를 목표로 한다.

- 브랜드 30개 이상
- 카테고리 6개
- 조건 태그 10개 이상
- 브랜드 상세 SEO 페이지
- 카테고리 페이지
- 태그 페이지
- 정보 수정 제보
- 관리자 브랜드/혜택 CRUD
- 검수 상태 관리
- 공식 출처 URL 관리
- 최근 확인일 표시
- 사이트맵/robots.txt

## 7. 하지 않는 것

MVP에서 하지 않는다.

- 사용자 회원가입
- 사용자 생일 입력 기반 개인화
- AI 추천
- WebSocket
- 알림톡
- 대량 자동 크롤링
- 브랜드 로고 무단 사용
- 앱 출시
- 카드/통신사/정부지원금 전체 비교
- 실시간 할인 정보 통합
